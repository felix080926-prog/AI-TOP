## 01 | 核心逻辑：轻量才是正义

**HKUDS/nanobot** 来自香港大学数据科学实验室，定位非常清晰：**Lightweight, open-source AI agent for your tools, chats, and workflows.**

它的核心哲学就一句话——**把你每天都在用的工具、聊天记录和工作流，交给一个轻量级的 AI Agent 来编排。**

nanobot 的架构逻辑可以拆解为三层：

1. **工具层**：连接你的真实工具（代码、API、数据库、浏览器等）
2. **推理层**：LLM 根据你的指令，理解意图并规划步骤
3. **执行层**：自动调用工具完成多步骤任务

关键区别在于：nanobot 没有一上来就搞一个庞然大物的框架。它选择了「模块化 + 可插拔」的路线，你可以只挂载需要的工具，按需扩展。这是它和 LangChain、AutoGPT 等重量级框架最本质的不同。

学术背景让它的设计更「干净」——代码结构清晰、文档完善、对新手友好。这在开源 Agent 项目中相当少见。

---

## 02 | 硬核功能盘点

> 🔗 GitHub 地址：https://github.com/HKUDS/nanobot

nanobot 的核心能力可以用四个关键词概括：

**1. 多工具聚合**
你不用再为每个工具单独写对接代码。nanobot 内置了工具适配器，一行配置就能接入代码执行、API 调用、文件读写、数据库查询等常见操作。

**2. 多轮对话编排**
支持基于上下文的多轮任务。你可以说「先帮我查一下这周的数据，然后根据数据写一个摘要，再发到飞书群里」——它会记住每一步的结果并递进执行。

**3. 热插拔扩展**
想加一个新工具？写一个简单的工具描述文件，nanobot 就能自动识别并纳入编排范围。不需要改框架代码，不需要重启服务。

**4. 本地优先**
轻量级的另一个含义是：可以在本地跑。不需要 GPU，不需要高配服务器。一个普通的开发机就能跑起来，这对个人开发者来说太友好了。

## 03 | 典型场景和避坑

**最适合的场景：**

- **个人效率自动化**：自动整理笔记、生成日报周报、跨工具数据搬运
- **小团队协作**：把 Slack/飞书消息、GitHub Issue、Notion 文档串联起来，实现半自动 workflow
- **快速 PoC 验证**：想测试某个 AI 工作流的可行性？用 nanobot 搭个原型，半小时出结果

**常见坑点：**

- **工具描述要准确**：nanobot 通过工具描述来理解它能做什么。描述写得太模糊，LLM 就不会正确调用。建议参考官方示例的格式
- **复杂依赖链需测试**：虽然多步骤编排很强大，但超过 5 步的复杂链路建议分段测试，避免中间环节出错导致整条链路重跑
- **本地模型精度有限**：如果用本地小模型做推理，复杂任务的准确率会下降。建议在重要场景中使用云端模型（如 DeepSeek、GPT）做推理层

## 04 | 上手教程

nanobot 的上手非常简单，前提是你有基本的 Python 环境和 API Key。

**第一步：安装**
```bash
pip install nanobot-ai
```

**第二步：配置工具**
创建一个 `tools.yaml` 文件，定义你要接入的工具：
```yaml
tools:
  - name: code_executor
    type: python
    description: "执行 Python 代码"
  - name: web_search
    type: api
    endpoint: "https://api.xxx.com/search"
    api_key: "${SEARCH_API_KEY}"
```

**第三步：启动 Agent**
```bash
nanobot start --tools tools.yaml
```

**第四步：开始对话**
在终端输入自然语言指令，nanobot 会自动分析意图、调用工具、返回结果。

整个流程从安装到跑通第一个任务，正常在 3 分钟内可以完成。这个体验在当前的 Agent 框架里可以说是最轻量的之一。

## 05 | 总结

nanobot 的意义不在于它比 LangChain 强在哪里，而在于它重新定义了「AI Agent 应该有多重」。

当这个领域被各种企业级框架堆得越来越重的时候，HKUDS 团队选择了一个完全相反的方向——**极致轻量、开箱即用、个人友好**。

如果你还在观望 AI Agent 到底能做什么，nanobot 应该是你第一个动手试的项目。
