**大家好，我是雷达君。**

你有没有算过一笔账：你的 AI 应用每次处理用户请求，都在重复计算同样的东西？

用户 A 问“帮我总结一下这份 500 页的合同”，AI 花 3 秒读完合同，输出摘要。用户 B 问“这份合同里关于违约责任的条款有哪些”，AI 又把那 500 页合同从头到尾读了一遍。

再算一笔更大的账：你部署了一套 RAG 系统，100 个用户同时提问，每个人都问的是同一份公司政策文档里的不同问题——AI 把这份文档的 KV 缓存重复计算了 100 遍。

**大模型推理最烧钱的地方，从来不是生成答案的那几秒钟，而是每次都要从头“阅读”一遍输入内容的 prefill 阶段。**

KV Cache（键值缓存）技术就是用来解决这个问题的——把模型“读过”的内容存下来，下次遇到相同内容直接复用，跳过重复计算。但传统 KV Cache 有个致命局限：它只存在于单个 GPU 的显存里，请求一结束就被丢弃。同一个文档被 100 个用户问 100 次，KV Cache 就被计算 100 次、丢弃 100 次——GPU 算力全浪费在重复劳动上。

今天雷达君要安利的这个开源项目，彻底打破了这道墙。它把 KV Cache 从 GPU 显存的“牢笼”里解放出来，让它可以跨请求、跨引擎、跨节点地被复用和共享。

它来自芝加哥大学的突破性研究，已被 PyTorch Ecosystem 正式收录。NVIDIA、AMD、Intel、Red Hat、微软、IBM、腾讯、字节跳动等 30 多家产业合作伙伴都在围绕它构建生态。

它就是——**LMCache**。

## 01 ｜核心逻辑：把 KV Cache 从“一次性用品”变成“共享资产”

LMCache 的核心理念，一句话就能说清：**每一段文本，模型只需推理一次**。

你每次问 AI 一个问题，AI 都要经历两个阶段：**Prefill（预填充）** ——把输入文本读进模型，计算 KV Cache；**Decode（解码）** ——基于 KV Cache 逐 token 生成答案。其中 Prefill 阶段占了推理延迟和算力消耗的大头。

KV Cache 技术的本质，就是把 Prefill 阶段计算出来的中间结果缓存下来。下次遇到相同或相似的输入，直接复用缓存，跳过 Prefill。

但传统 KV Cache 有一个致命问题：它被锁在单个 GPU 的显存里，请求结束就被丢弃。

LMCache 的做法是**把 KV Cache 从 GPU 显存里“捞”出来**，存到更便宜、更大容量的存储层——CPU 内存、本地 SSD、甚至分布式存储和 S3 对象存储。当新请求到来时，LMCache 先在这些存储层里查找是否已有可复用的 KV Cache，有就直接用，没有才让模型重新计算。

这样带来的效果是：

- 对于多轮对话：历史对话的 KV Cache 被持久化，同一会话的后续轮次直接复用，TTFT（首 Token 延迟）下降**3-10 倍**。
- 对于 RAG 场景：同一份文档被不同用户提问，只需计算一次 KV Cache，后续全部复用。
- 对于 Agent 工作流：多个 Agent 共享同一份上下文，不用各自重新计算。

**一句话：别人的 KV Cache 是“一次性用品”，用完就扔；LMCache 让 KV Cache 变成“共享资产”，所有请求、所有引擎、所有节点都能复用。**

## 02 ｜硬核功能盘点

> 🔗 **GitHub 地址**：https://github.com/LMCache/LMCache

**1. 分层存储：从 GPU 显存到 S3，缓存容量扩大 100 倍**

LMCache 最核心的能力，是一套**三级缓存架构**：

- **L1（GPU 显存）** ：纳秒级访问，存最高频的 KV Cache
- **L2（CPU 内存 + 本地 SSD）** ：微秒级访问，存次高频的 KV Cache
- **L3（分布式存储 / 对象存储 / S3）** ：毫秒级访问，存所有历史 KV Cache，容量近乎无限

传统方案只能在 GPU 显存里缓存，容量受限。LMCache 把缓存容量扩大了**100 倍以上**——只要你愿意，可以把过去半年所有用户对话的 KV Cache 全存下来。Google Cloud、CoreWeave 等云厂商已经基于这套分层存储架构，在 GKE 上实现了生产级部署。

**2. 跨引擎共享：vLLM、SGLang、NVIDIA Dynamo 全打通**

LMCache 不是某个推理引擎的“私有插件”，而是一个**独立的 KV Cache 管理层**。它原生支持 vLLM 和 SGLang 两大主流推理引擎，并且已经集成到 NVIDIA Dynamo、vLLM Production Stack、KServe 等开源项目中。你在 vLLM 里生成的 KV Cache，可以在 SGLang 里复用——这在以前是不可能的。

> 截至 2026 年 6 月，LMCache 社区刚刚宣布获得重大产业投资支持，AMD、NVIDIA 和 CoreWeave 均参与了战略投资。这意味着从 GPU 厂商到云厂商，整个产业链都在押注“KV Cache 独立层”这个方向。

**3. 3-10 倍 TTFT 降低 + 最高 15 倍吞吐提升**

LMCache 的性能数据非常硬核。与 vLLM 结合使用时：

- 多轮问答和 RAG 场景下，**TTFT 降低 3-10 倍**
- 吞吐量最高提升**15 倍**
- Agentic 工作负载下（32 用户/100K 上下文）：TTFT 平均降低**3 倍**，P95 降低**2.1 倍**，最大延迟降低**2.6 倍**，请求处理能力提升**2.3 倍**
- 缓存命中率提升**60-80%**，单请求延迟降低**40-70%**，显存占用减少**30-50%**

对于多模态模型，LMCache 同样有效——重复图像的 KV Cache 命中率接近**100%**，端到端延迟从数十秒压缩到约**1 秒**。

**4. PD 分离：Prefill 和 Decode 解耦，GPU 利用率最大化**

这是 LMCache 在架构层面最前沿的能力。传统推理中，Prefill（计算密集）和 Decode（内存密集）在同一个 GPU 上串行执行，GPU 要么在算、要么在等，利用率很低。LMCache 支持**PD 分离（Prefill-Decode Disaggregation）** ——把 Prefill 和 Decode 分配到不同的 GPU 集群上执行，通过 KV Cache 的跨节点传输实现流水线并行。这意味着你可以把昂贵的 H100 专门用来跑 Prefill，把相对便宜的 GPU 用来跑 Decode，整体 GPU 利用率大幅提升。

> _现实摩擦提醒_：PD 分离需要多节点网络支持（如 RDMA），对基础设施有一定要求。但在 NVIDIA Dynamo 等生产级堆栈中，LMCache 已经提供了开箱即用的支持。

**5. MP 模式：KV Cache 管理独立于推理进程**

LMCache v1.0 引入的**MP（Multi-Process）模式**，把 KV Cache 的管理从推理引擎进程中彻底剥离出来，作为一个**独立进程**运行。所有 GPU Worker 通过 RPC 与 LMCache Server 通信，共享统一的 KV Cache 池。这个架构带来的好处是：推理引擎升级不影响缓存层；多个推理实例共享同一份缓存；缓存容量不再受单个 GPU 显存限制。

**6. 220+贡献者 + PyTorch Ecosystem 官方收录**

2024 年 6 月从芝加哥大学的实验室里诞生，一年后成为 PyTorch Ecosystem 官方项目。超过 220 位贡献者来自全球 30 多家机构。这个社区的广度和深度，保证了 LMCache 不会因为某个公司的战略调整而停摆。

## 03 ｜横向对比

在 LLM 推理加速领域，LMCache 的定位和传统 KV Cache 方案完全不同：

| 维度             | **LMCache**                        | **vLLM 原生 Prefix Caching** | **Hugging Face Transformers** |
| :--------------- | :--------------------------------- | :--------------------------- | :---------------------------- |
| **缓存范围**     | **跨请求、跨引擎、跨节点**         | 单实例内                     | 单次推理内                    |
| **存储介质**     | GPU 显存 →CPU 内存 →SSD→S3（三级） | 仅 GPU 显存                  | 无持久化                      |
| **缓存生命周期** | **持久化（天/周/月）**             | 请求结束即丢弃               | 请求结束即丢弃                |
| **引擎兼容性**   | vLLM、SGLang、Dynamo 等            | 仅 vLLM                      | 仅 Transformers               |
| **PD 分离支持**  | ✅ 原生支持                        | ❌                           | ❌                            |
| **典型场景**     | 生产级高并发、RAG、多轮对话        | 单实例轻量推理               | 研究/单次推理                 |

简单说：**vLLM 原生 Prefix Caching 是“单机缓存”，LMCache 是“数据中心级的缓存基础设施”。**

## 04 ｜典型场景和避坑

**适合谁用？**

- **在生产环境运行高并发 LLM 服务（RAG、多轮对话、Agent）的团队**：LMCache 能让相同或相似的输入不再重复计算，TTFT 和 GPU 成本双降。
- **需要跨多个推理引擎共享缓存的平台团队**：vLLM 和 SGLang 的 KV Cache 可以互通，不用重复建设。
- **对数据主权有要求、希望缓存持久化的企业**：KV Cache 可以存到自有的 S3 或分布式存储中，不依赖第三方。
- **正在探索 PD 分离架构的先进团队**：LMCache 是目前开源社区里 PD 分离最成熟的方案之一。

**不推荐给谁？**

- **单次推理、无重复输入的轻量场景**：加一层缓存中间件属于过度设计，直接 vLLM 就够了。
- **完全没有 Python 和容器化经验的团队**：LMCache 的部署需要一定的技术基础，虽然官方提供了详细的 Docker 和 K8s 部署指南。
- **只用 CPU 推理、不用 GPU 的场景**：LMCache 主要面向 GPU 推理优化，CPU 场景收益有限。

## 05 ｜快速上手指南

LMCache 与 vLLM 的集成极其简单，**改动一行代码就能体验效果**。

**方式一：vLLM + LMCache（最推荐）**

```bash
# 安装
pip install lmcache vllm

# 启动带LMCache的vLLM服务（只需替换vllm serve为lmcache_vllm serve）
lmcache_vllm serve Qwen/Qwen3-4B-Instruct \
  --kv-transfer-config '{"kv_connector":"LMCacheConnectorV1", "kv_role":"kv_both"}'
```

就这么简单。启动后，所有 KV Cache 会自动被 LMCache 管理和持久化。

**方式二：配置 LMCache 独立服务（适合生产环境）**

创建配置文件`lmcache.yaml`：

```yaml
lmcache:
  local_cache_size: 10GB
  remote_cache: redis://localhost:6379
  storage_backend: local_disk
```

启动 LMCache 服务：

```bash
lmcache_server --config lmcache.yaml
```

然后在 vLLM 启动时指定 LMCache 连接器即可。

**方式三：Kubernetes 生产部署（适合大规模集群）**

LMCache 官方提供了完整的 Helm Chart 和 DaemonSet 部署方案。在生产环境中，建议：

- 每个 GPU 节点部署一个 LMCache DaemonSet
- 用 Redis 或 S3 作为跨节点的共享缓存层
- 持续监控缓存命中率，根据业务特点调整缓存策略

## 06 ｜总结

LMCache 解决的问题很朴素，但影响很深远：**大模型推理中，重复计算太多，GPU 算力被严重浪费。**

在 LLM 推理场景里，真正昂贵的不是生成答案的 Decode 阶段，而是每次都要从头“阅读”输入内容的 Prefill 阶段。LMCache 用一套独立于推理引擎的 KV Cache 管理层，把 Prefill 的结果持久化、共享化、跨节点化——让 GPU 算力只花在真正需要的地方，而不是一遍又一遍地重复计算同样的内容。
