**大家好，我是雷达君。**

你有没有过这样的经历：Claude 的 API 额度用完了，GPT 的账单已经飙到了三位数，DeepSeek 的 key 还藏在某个 .env 文件里找不到。你打开五个网页，切来切去，就为了找一个还能用的模型。写代码十分钟，管 API key 一小时——这不叫搞技术，这叫当会计。

更让人崩溃的是，你以为自己用的是"最便宜的模型"，结果月末账单告诉你，一个不留意，上下文膨胀了十倍。token 跟流水一样哗哗往外淌，可你连到底该用哪个模型、哪个最省钱都搞不清楚。

这其实是当下 AI 开发者的一个集体困境：模型越来越多、越来越强，但管理它们的基础设施远远没跟上。我们不是在"用 AI"，是在"伺候 AI"。

而 **OmniRoute**，用一个端点的体量，试图终结这种混乱。

它的核心理念极其粗暴：既然有 271 家 AI 供应商、500 多个模型摆在那，为什么不做一个智能调度中心，让你无感切换？更狠的是——它免费，MIT 协议开源，每月光免费 token 就堆了 **14 亿**（约 1.4B）。

## 01 ｜核心逻辑

OmniRoute 本质上是一个**智能 AI 网关**。你在 Claude Code、Cursor、Codex 这些 IDE 里正常写代码，OmniRoute 在背后帮你做一件事：自动把请求路由到当前最合适（最便宜、最快、额度最充裕）的模型上。

它的路由引擎叫 **Auto-Combo**，每次请求到来时，引擎会实时给每一个候选模型打 12 个维度的分数——健康度、剩余额度、单价、延迟、成功率、数据新鲜度等等——然后动态决定"这一轮让谁来回答"。如果当前模型突然挂了或者额度烧完，请求会自动滑向下一个，整个过程对你是完全无感的。

这背后还有一套叫 **Quota-Share** 的额度共享机制。比如你的团队买了一个 Codex Pro 订阅，正常情况是大家抢着用，一个人跑个大型重构，5 小时额度直接烧干，剩下的人干瞪眼。Quota-Share 按权重分配额度——给你 50%，给他 30%，给她 20%，而且如果有人在摸鱼，他的剩余额度会自动借给忙的那个，真正做到了"不浪费一滴额度"。

更夸张的是 **token 压缩**。OmniRoute 内置了两套压缩引擎协同工作：RTK 摘掉冗余上下文，Caveman 用极致压缩算法缩减 token。叠加起来，实测能省 15% 到 95% 的 token。这意味着你写一个原本要烧 10,000 token 的复杂函数，经过压缩可能只需要 1,500 个——速度更快、成本更低，关键是你的 Claude 额度终于能撑过一整个下午了。

## 02 ｜硬核功能盘点

> GitHub 地址：https://github.com/diegosouzapw/OmniRoute

- **271 家供应商、90+ 免费商、500+ 模型**：从 Claude 到 GPT，从 Gemini 到 Kimi K3，从 GLM 到 DeepSeek，一个端点全部打通。90 多家提供免费额度，每月累计约 14 亿 token 的"免费午餐"。
- **18 种路由策略**：优先级、加权、轮询、成本优化、延迟优先、额度感知、上下文接力——甚至可以启动 Fusion 模式，把同一个问题同时扔给多个模型，再让一个"裁判模型"综合最优答案。
- **RTK + Caveman 双压缩引擎**：不仅是压缩，还有内置的 Compress Studio 让你可视化拖拽调整压缩参数。中文文言文（wényán）都有专门的压缩包。
- **MCP 服务器（104 个工具）**：原生支持 MCP 协议，3 种传输方式、31 个权限范围。这意味着你可以在任何 MCP 客户端里直接把 OmniRoute 当工具链用。
- **A2A 代理协议**：6 个技能 + JSON-RPC 2.0，支持 agent-to-agent 通信，让多个 AI Agent 之间进行协作。
- **内存系统（FTS5 + 向量检索）**：会话记忆持久化，默认关闭、按需开启，支持 int8 量化和时间衰减，尊重隐私。
- **安全护栏**：每条 LLM 请求都经过 prompt 注入检测、PII 脱敏和视觉内容审查，还有免费的 DuckDuckGo 搜索作为"最后一手"。
- **全平台覆盖**：Web 仪表盘、桌面客户端、PWA、Termux 移动端——43 种语言界面。
- **云代理兼容**：原生对接 Codex、Cursor、Devin、Jules 等云 IDE 代理，一条命令完成配置。

## 03 ｜典型场景和避坑

**典型场景一：个人开发者想薅免费额度**

你不是重度用户，就是偶尔需要 AI 帮忙改个 bug、写个脚本。但 OpenAI 的免费额度一会就没了，Claude 又限制次数。装一个 OmniRoute，把所有免费 provider 配进去，设置路由策略为 `auto/cheap`（最便宜优先）——系统自动在所有免费商之间切换，你永远有额度可用。不少开发者反馈一个月能省下 $50-200 的 API 费用。

**典型场景二：小团队共享一个付费订阅**

团队 5 个人，只有一份 Claude Pro 订阅。之前靠微信群"还在用吗？我用完了没？"手动协调，效率极低。部署 OmniRoute + Quota-Share，按人头分配额度（产品经理 10%、前端 40%、后端 40%、QA 10%），每个人的消耗实时可见，超额自动切到备用模型。会议室的"额度焦虑"消失了。

**避坑指南**

- 第一，免费商不一定是"完全免费"——有些需要注册拿 API key（如 Google Gemini），有些是限速的（如某些 HuggingFace 推理端点）。部署前先仔细读一遍 `/dashboard/free-tiers` 的详细说明，别以为装了就万事大吉。
- 第二，压缩不是万能的——RTK+Caveman 的 15-95% 节省率取决于你的 prompt 结构。如果你本身就是精简 prompt 的用户，压缩收益可能在 20% 左右；但如果你习惯把整本手册塞进 context，那 90%+ 的压缩率会让你拍案叫绝。建议先跑一周记录日志，再决定是否开启全量压缩。
- 第三，Quota-Share 需要连接池管理——如果你把所有请求都挤在同一个连接上，即使额度分配得很公平，并发瓶颈也会导致部分用户等待。建议为每个成员分配独立的 connection 配置。

## 04 ｜上手教程

用 Docker 部署最快，三行命令起飞：

```bash
# 拉取镜像
docker pull ghcr.io/diegosouzapw/omniroute:latest

# 启动服务（暴露 8080 端口）
docker run -d --name omniroute \
  -p 8080:8080 \
  -v ~/omniroute-data:/app/data \
  ghcr.io/diegosouzapw/omniroute:latest
```

然后打开 `http://localhost:8080`，进入仪表盘，在 Providers 页面逐个添加你的 API key（Claude、GPT、Gemini 等）。每个 provider 配好之后，系统会自动纳入路由池。

接下来配置你的 IDE。以 Cursor 为例，在设置里把 API Base URL 改成 `http://localhost:8080/v1`，API Key 填你在 OmniRoute 管理后台生成的 key（不是 Claude 的原始 key），Model 选 `auto`。保存之后，你的每一次 AI 请求都会经过 OmniRoute 的智能路由。

如果你用的是 Claude Code 或 Codex CLI，OmniRoute 官方提供了 `setup-*` 系列命令，一条指令自动完成配置：

```bash
# 自动配置 Claude Code
npx omniroute setup-claude-code

# 自动配置 Codex
npx omniroute setup-codex
```

想验证路由是否生效？发一个请求后在仪表盘 `Logs` 页面查看——你会看到每次请求实际路由到了哪个模型、消耗了多少 token、花了多少钱。

## 05 ｜总结

OmniRoute 解决的不是"模型不够好"的问题，而是"模型太多、管不过来了"的问题。在当下这个每天都有新模型开源的节奏里，它提供的是一种"以不变应万变"的心态：你只管写代码，路由的事交给我。

当然，它也有限制。免费商家的稳定性参差不齐，有些小供应商可能今天还在、明天就跑路了。Quota-Share 的分发精度在极端场景（比如启动瞬间所有成员同时发请求）下会有几秒的延迟。压缩引擎虽然强悍，但对某些特殊格式（如结构化 JSON schema）需要手动微调参数。

但这些都不妨碍它是目前开源界最全面的 AI 网关方案。500 多位贡献者、5,400 多次提交、13 分钟内还在打 hotfix——这个社区的热度本身就说明了一切。如果你正在被 API 账单或者多模型管理折磨，今天就可以把 OmniRoute 拉下来试试。最坏的情况不过是 Docker 多占了几百兆磁盘，最好的情况是——你的 API 账单真的腰斩了。
