**大家好，我是雷达君。**

一个 Claude Code 代理跑得好好的。两个一起跑也还行。但当你同时在同一个代码库上跑六个 AI 代理——一个在改登录模块、一个在重构支付逻辑、一个在修样式 Bug——你开始慌了。

谁在改什么？哪个 worktree 承载着改动？代理中途崩溃了怎么办？上下文丢了谁来补？

这是今天很多 AI 驱动开发团队正在经历的噩梦：**Agent 越多，效率反而越低。**

前 Google、Amazon 资深工程师 Steve Yegge 也遇到了这个问题。他的解法不是让 AI 更聪明，而是给它们配了一套**“操作系统”**——一个能同时协调 20-30 个 AI 代理的工作空间管理器。

它就是——**Gas Town**。

## 01 ｜核心逻辑：AI 代理的“城市操作系统”

Gas Town 把自己定位为一个**多代理工作空间管理器**。但更准确的说法是：它是**AI 代理的操作系统**。

想象一下：你是一座“Gas Town”的镇长。镇子里有多个“Rig”（项目容器），每个 Rig 包装了一个 Git 仓库。Rig 里有“Polecats”——这些是临时性的工作代理，需要时生成，任务完成就消失，但身份和工作历史会永久保留。

每个 Rig 都配有“Hooks”——基于 git worktree 的持久化存储。代理崩溃了？重启后接着干。上下文丢了？Hooks 里全在。

Gas Town 把 AI 代理从“用完就忘的临时工”，变成了**有身份、有记忆、可追踪的正式员工**。

## 02 ｜硬核功能盘点

> 🔗 **GitHub 地址**：https://github.com/gastownhall/gastown

### 1. The Mayor：你的 AI“镇长”

Mayor 是一个 Claude Code 实例，掌握着你工作空间、项目和代理的全部上下文。

**你不需要跟每个代理沟通。你只需要告诉 Mayor 你想做什么，剩下的由它协调。**

### 2. 持久化工作追踪：代理崩溃？不怕

这是 Gas Town 最核心的价值。传统的 AI 代理一旦重启，之前的工作上下文就全丢了。

Gas Town 通过 git-backed hooks 把所有工作状态持久化。代理崩溃了、重启了、换人了——**工作不会丢，下一个代理可以接着干**。

### 3. 规模化编排：从 4 个到 30 个

没有 Gas Town 的时候，4-10 个代理就已经乱成一团了。有了 Gas Town，你可以**轻松扩展到 20-30 个代理同时工作**。

### 4. Beads + Convoys：工作追踪的“账本系统”

Gas Town 集成了 Beads——一个基于 Git 的图状问题追踪系统。每个工作项都有一个唯一 ID（如`gt-abc12`）。Convoys 则把多个 Beads 打包分配给代理。

**你不再是“猜”谁在做什么，而是在“看”一个清晰的工作账本。**

### 5. Molecules：标准化工作流模板

Molecules 是预定义的 TOML 工作流模板。复杂任务不再需要每次都重新规划——定义一次，反复使用。

### 6. 三层监控：Witness、Deacon、Dogs

Gas Town 有一套三层的看门狗系统：

- **Witness**：每个 Rig 的生命周期管理器，监控 Polecats、检测卡住的代理、触发恢复
- **Deacon & Dogs**：更高层级的健康监控，确保整个系统平稳运行

### 7. 多代理兼容：Claude Code、Copilot、Codex、Gemini……

Gas Town 不挑食。它支持 Claude Code、GitHub Copilot、Codex、Gemini 等主流 AI 编码代理。你想用什么就用什么。

## 03 ｜和同类项目有什么不同？

Gas Town、Herdr 和 OpenClaw 都在做“多代理管理”，但切入点完全不同：

| 维度           | **Gas Town**                    | **Herdr**                    | **OpenClaw**         |
| :------------- | :------------------------------ | :--------------------------- | :------------------- |
| **核心定位**   | AI 代理操作系统                 | AI 代理终端多路复用器        | AI 代理万能连接器    |
| **解决的问题** | 代理工作状态丢失、协调混乱      | 看不到代理状态、无法远程接入 | AI 无法接入多渠道    |
| **状态持久化** | ✅ Git-backed hooks             | ❌ 依赖终端会话              | ❌ 无                |
| **工作追踪**   | ✅ Beads 账本 + Convoys         | ❌                           | ❌                   |
| **规模化能力** | 20-30 个代理                    | 取决于终端窗口               | 取决于渠道数         |
| **典型用户**   | 需要大规模 AI 代理协作的团队    | 同时跑多个 Agent 的开发者    | 极客、多渠道部署用户 |
| **创始人背景** | Steve Yegge（前 Google/Amazon） | 个人开发者                   | 社区驱动             |

**简单说：Herdr 让你“看见”代理，OpenClaw 让代理“连接”世界，Gas Town 让代理“有组织、有记忆、可扩展”。**

## 04 ｜典型场景和避坑

**适合谁用？**

- **同时跑多个 AI 代理、需要协调它们工作的开发团队**：Gas Town 能让你从“手忙脚乱”变成“从容调度”
- **需要长周期 AI 自动化任务的场景**：代理崩溃不怕，Hooks 里的工作状态不会丢
- **想系统化 AI 驱动开发流程的团队**：Beads + Convoys + Molecules 提供了完整的工作追踪和流程标准化能力

**不推荐给谁？**

- **只用单个 AI 代理的个人开发者**：杀鸡用牛刀
- **完全不用终端、不用命令行的纯产品岗位**：Gas Town 的核心操作在 CLI 里

## 05 ｜快速上手指南（3 分钟）

**第 1 步：安装**

```bash
# Homebrew（推荐）
brew install gastown

# 或通过npm
npm install -g @gastown/gt

# 或从源码
go install github.com/steveyegge/gastown/cmd/gt@latest
```

**第 2 步：初始化工作空间**

```bash
gt install ~/gt --git
cd ~/gt
```

**第 3 步：添加项目**

```bash
gt rig add myproject https://github.com/you/repo.git
```

**第 4 步：创建个人工作空间并启动 Mayor**

```bash
gt crew add yourname --rig myproject
cd myproject/crew/yourname
gt mayor attach
```

**验证安装成功**：运行`gt --version`，如果返回版本号，说明安装成功。

## 06 ｜总结

Gas Town 解决的核心问题是：**AI 代理越多，管理越乱。**

一个代理很听话，两个还凑合，四到十个就开始互相踩脚。上下文丢了、工作状态丢了、谁在改什么完全看不清。

前 Google、Amazon 资深工程师 Steve Yegge 的解法不是让 AI 更聪明，而是给 AI 配了一套**“操作系统”**——Mayor 负责协调、Polecats 负责执行、Hooks 负责持久化、Beads 负责追踪、Convoys 负责打包、Molecules 负责标准化。

安装包和完整代码都在 GitHub 上：https://github.com/gastownhall/gastown
