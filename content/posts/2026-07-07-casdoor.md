**大家好，我是雷达君。**

你做了一个 SaaS 产品，用户用邮箱密码登录。后来用户说"我要用 Google 登录"，你加上了 Google OAuth。再后来企业客户说"我们要用 SAML 对接公司内部 SSO"，你又加上了 SAML。紧接着，甲方说"我们要多租户隔离"，你的数据库开始加`tenant_id`字段。再然后，审计要求出来了，你要记录每一次登录、每一次权限变更……

你算了一下，光是"用户身份"这件事，已经吃掉你**两个月**的开发时间。

更麻烦的是，你做的这一切，在下一个新项目里还要**再来一遍**。

这个问题的本质是：**每一个需要用户的应用都需要一套身份系统，但每一个团队都在重复发明同一个轮子。**

今天雷达君要安利的这个开源项目，就是来终结这种重复劳动的。它是一个**AI 原生的开源身份与访问管理（IAM）平台**——OAuth、OIDC、SAML、LDAP、MFA、WebAuthn、SCIM、多租户、RBAC……你需要的所有身份能力，全部开箱即用。

更重要的是，它生来就是为了 AI Agent 时代设计的——内置**MCP 网关**和**A2A 协议**支持，让 AI Agent 也能安全地完成身份认证和权限管理。

它就是——**Casdoor**。

## 01 ｜核心逻辑：身份认证，不应该成为 SaaS 开发的瓶颈

Casdoor 的核心理念可以用一句话概括：**把身份认证从"需要专门团队维护的基础设施"，变成"一行配置就能用的平台能力"。**

它由**Casbin**（全球最流行的开源权限管理库）团队打造，继承了 Casbin 在权限管理领域多年的工程积累。但 Casdoor 不只是"加了 UI 的 Casbin"——它是一个完整的、生产就绪的**身份与访问管理服务器**。

传统的身份管理方案是什么样？Auth0 功能强但闭源且昂贵，Keycloak 功能全但配置复杂得像在拆炸弹，Okta 面向大企业但个人开发者根本用不起。

Casdoor 走了一条不一样的路：**开源、自托管、UI 优先、AI 原生**。

它把所有身份能力打包成一个**开箱即用的平台**——你部署一个 Casdoor 实例，在 Web 控制台里配置应用、组织、用户、身份提供商，然后通过几行 SDK 代码接入你的应用。不需要自己设计用户表、不需要自己实现 OAuth 流程、不需要自己写权限校验逻辑。

**但 Casdoor 真正拉开差距的地方，是它对 AI Agent 时代的深度拥抱。**

> **现实摩擦提醒**：Casdoor 的功能非常全面，但这也意味着初次接触时面对的功能选项较多。建议先通过官方 Demo（`door.casdoor.com`）体验核心流程，再逐步深入高级功能。

## 02 ｜硬核功能盘点

> 🔗 **GitHub 地址**：https://github.com/casdoor/casdoor

### 1. MCP 网关 + A2A 协议：AI Agent 的"身份证系统"

这是 Casdoor 在 2026 年最值得关注的能力升级，也是它和传统 IAM 工具最本质的区别。

随着 MCP（模型上下文协议）的普及，AI Agent 正在大量接入外部工具和数据源。但一个问题随之而来：**AI Agent 怎么安全地完成身份认证和权限管理？** 传统 IAM 工具根本没考虑过这个场景。

Casdoor 的答案是：**内置 MCP 网关**。任何一个支持 MCP 的 AI 客户端——Claude Desktop、Cursor、Windsurf、VS Code + GitHub Copilot——都可以通过 Casdoor 的 MCP 端点完成身份认证。

配置方式极其简单，在 MCP 客户端配置文件中添加 Casdoor 端点即可：

```json
{
  "mcpServers": {
    "casdoor": {
      "url": "https://your-casdoor.com/api/mcp",
      "headers": { "Authorization": "Bearer <your-token>" }
    }
  }
}
```

更硬核的是，Casdoor 还支持**A2A（Agent-to-Agent）协议**——Agent 之间可以安全地互相通信和授权。这意味着你的 AI Agent 们不仅知道"你是谁"，还能知道"你能做什么"。

**你的 AI Agent 终于有了一套完整的身份体系。**

### 2. 原生 OpenClaw 集成：AI Agent 的可观测性

如果你在用 OpenClaw（就是那个 Star 数 37 万的"龙虾"），Casdoor 可以直接接收 OpenClaw 的遥测数据。

OpenClaw 会把每一次工具调用、Agent 间交互、LLM 调用的轨迹推送到 Casdoor，以**结构化条目**的形式存储下来。你可以在 Casdoor 的 Web UI 里查看完整的 Span 树，实时追踪你的 AI Agent 到底在执行什么操作。

配置同样极其简单：

```yaml
exporters:
  otlphttp:
    endpoint: https://your-casdoor.com
    headers:
      Content-Type: application/x-protobuf
```

**你的 AI Agent 不再是一个"黑盒"，而是一个可以被审计、被追踪、被管理的系统组件。**

### 3. 协议全家桶：OAuth、OIDC、SAML、CAS、LDAP……一个不落

Casdoor 支持的身份协议清单长得像一本百科全书：

- **OAuth 2.0 / OIDC**：业界标准的授权和身份认证协议
- **SAML 2.0**：企业 SSO 集成，对接 Okta、Azure AD 等
- **CAS**：中央认证服务
- **LDAP**：目录服务集成
- **WebAuthn / Passkeys**：无密码认证
- **TOTP / MFA**：多因素认证
- **Face ID**：生物识别认证

**不管你对接什么系统，Casdoor 都能"翻译"成统一的身份语言。**

### 4. 企业级能力：多租户、SCIM、RBAC、审计日志

对于 B2B SaaS 场景，Casdoor 提供了完整的企业级能力矩阵：

- **SCIM 2.0**：跨系统用户自动 provisioning
- **RBAC**：基于角色的访问控制
- **多租户**：组织级别的数据隔离
- **审计日志**：每一次登录、权限变更全程可追溯
- **Social Login**：Google、GitHub、Azure AD 等 100+身份提供商

### 5. UI 优先 + 完整开发者体验

Casdoor 把自己定位为一个**UI-first**的身份平台。你不需要对着配置文件改到怀疑人生，所有管理操作——用户管理、应用配置、组织管理、身份提供商配置——都在一个现代化的 Web 控制台里完成。

对于开发者，Casdoor 提供了完整的 RESTful API、Swagger UI 交互式文档、以及覆盖 Go、Java、Python、Node.js 等主流语言的 SDK。同时支持 Webhooks 事件驱动集成和品牌主题定制。

### 6. 技术栈：Go + React，可水平扩展

Casdoor 采用前后端分离架构：

- **后端**：Go 语言 + Beego 框架，RESTful API
- **前端**：React，现代化 Web UI
- **数据库**：支持 MySQL、PostgreSQL 等主流数据库
- **缓存**：可选 Redis

这种架构让 Casdoor 可以水平扩展，支持高并发场景。

## 03 ｜深度对比：Casdoor vs Logto

之前，雷达君向大家推荐过[Logto](https://mp.weixin.qq.com/s/dauH24iF2ncn8qNiPWdrxw)。在 AI 身份认证这个新兴赛道上，Casdoor 和 Logto 是差不多同级别的两个顶尖开源项目，经常被放在一起比较。它们没有谁碾压谁，而是代表了两种不同的设计哲学。

### ⚔️ Casdoor vs Logto：核心对比

| 维度            | **Casdoor**                                 | **Logto**                                        |
| :-------------- | :------------------------------------------ | :----------------------------------------------- |
| **核心定位**    | **AI 优先（Agent-first）** 的 IAM 平台      | **开发者优先（Developer-first）** 的身份基础设施 |
| **技术栈**      | **Go** + React                              | **TypeScript**（Node.js）+ React                 |
| **开源协议**    | Apache 2.0                                  | MPL 2.0                                          |
| **协议支持**    | **最全**：OAuth/OIDC、SAML、CAS、LDAP、SCIM | 核心：OIDC、OAuth 2.1、SAML                      |
| **AI 原生特性** | **内置 MCP Server、A2A 协议支持**           | 提供独立的 Logto MCP Server                      |
| **权限模型**    | **Casbin**（支持 ACL/RBAC/ABAC 等）         | 原生组织级 RBAC                                  |
| **UI / UX**     | UI 优先，功能全面                           | 体验现代化，开箱即用的登录界面                   |
| **云服务**      | Casdoor Cloud                               | Logto Cloud（SOC 2 Type II 认证）                |
| **目标用户**    | 需要复杂企业集成+AI 原生架构的团队          | 追求极致开发者体验的 JS/TS 团队                  |

### 🔍 关键差异解读

**1. AI 时代的入场姿势不同**

两者都拥抱 AI，但方式完全不一样。

Logto 在 2026 年 2 月正式推出了独立的**Logto MCP Server**，让你的 AI 工具用一句 Prompt 就能完成 Logto 的配置和集成。这是一个非常优雅的"插件式"方案——AI 能力是独立扩展出来的。

Casdoor 则是直接把**MCP 服务器内嵌到核心架构里**。AI Agent 可以通过自然语言直接与 Casdoor 交互，管理用户和权限。它还支持**A2A（Agent-to-Agent）协议**，让 Agent 之间也能安全通信。这是一个"内嵌式"的 AI 原生方案。

从路线选择上看：Logto 说"让 AI 更容易使用身份系统"，Casdoor 说"让身份系统本身成为 AI 生态的一部分"。前者更实用，后者更激进。

**2. 技术栈决定了性格**

这是两者最根本的区别。Casdoor 选用 **Go**，天生适合构建高并发、资源占用低的后端服务，特别契合它作为"身份中台"的定位。Logto 基于**TypeScript/Node.js**，与 Next.js、React 等现代前端技术栈同源，对广大 JS/TS 开发者来说，上手和二次开发的门槛更低。

**简单说：用 Go 的团队会更倾向 Casdoor，用 JS/TS 的团队会更倾向 Logto。**

**3. "功能全面" vs "体验纯粹"**

Casdoor 支持**CAS、LDAP、SCIM**等更广泛的企业级协议，权限模型通过**Casbin**实现，极其灵活，支持 ACL、RBAC、ABAC 等多种模式。但这种"大而全"也让它略显复杂，初次接触时功能选项较多。

Logto 则更聚焦，核心围绕 OIDC/OAuth 2.1 和 SAML，把多租户 RBAC 等核心功能做得非常精致和现代化。它的登录界面开箱即用，开发者体验是其核心卖点。

## 05 ｜典型场景和避坑

**适合谁用？**

- **正在构建 SaaS 产品、需要完整身份认证和权限管理的开发团队**：Casdoor 把多租户、SSO、MFA、RBAC 全部开箱即用
- **需要对接企业 SSO（SAML/OIDC）的 B2B SaaS**：企业客户要求用 Okta 或 Azure AD 登录？Casdoor 原生支持
- **正在构建 AI Agent 或 MCP 应用、需要安全认证机制的开发者**：Casdoor 的 MCP 网关让 AI Agent 有了"身份证系统"
- **以 Go 为主要技术栈的后端团队**：Casdoor 的 Go 生态原生集成
- **不想被 Auth0 等闭源方案锁定的技术团队**：完全开源、完全自托管
- **需要从 Keycloak 迁移到更现代方案的老项目**：Casdoor 提供了更清爽的 UI 和更清晰的配置方式

**如果以下场景更符合你的需求，Logto 可能是更好的选择**：

- 你的技术栈是**JavaScript/TypeScript**生态，追求同语言的全栈体验
- 你追求**极致现代化的开发者体验**和开箱即用的登录界面
- 你计划使用官方提供的**SOC 2 Type II 认证**的云服务
- 你不需要 CAS、LDAP 等更广泛的企业级协议

**不推荐给谁？**

- **只需要极简邮箱密码登录的个人项目**：用 Casdoor 属于杀鸡用牛刀
- **完全不想维护任何基础设施的非技术团队**：虽然 Casdoor 有 Docker 一键部署，但毕竟是一个需要维护的服务
- **已经深度绑定 Auth0 或 Okta 生态且迁移成本过高的项目**

## 06 ｜快速上手指南（5 分钟）

Casdoor 提供了多种部署方式，从"Docker 一键"到"源码编译"全覆盖。

### 方式一：Docker Compose（最推荐）

```bash
# 克隆仓库
git clone https://github.com/casdoor/casdoor.git
cd casdoor

# 使用Docker Compose一键启动
docker compose up
```

等待容器启动后，访问`http://localhost:8000`，使用默认账号`built-in/admin` / `123`登录。

> ⚠️ **生产环境务必立即修改默认密码！**

### 方式二：源码编译（适合深度定制）

```bash
# 安装依赖：Go 1.25+、Node.js LTS、Yarn 1.x
git clone https://github.com/casdoor/casdoor.git
cd casdoor

# 配置数据库（修改conf/app.conf）
# 构建前端
cd web
yarn install
yarn build
cd ..

# 启动后端
go run main.go
```

### 方式三：Kubernetes Helm（适合 K8s 环境）

```bash
helm install casdoor oci://registry-1.docker.io/casbin/casdoor-helm-charts
```

### 验证安装成功

访问`http://localhost:8000`，如果能正常打开登录页面并用默认账号登录，说明安装成功。登录后进入管理控制台，创建一个测试应用并完成 SDK 集成即可。

## 07 ｜总结

Casdoor 解决的核心问题是什么？

**每一个需要用户的应用都需要一套身份系统，但每一个团队都在重复发明同一个轮子。**

邮箱密码登录、社交登录、企业 SSO、多租户隔离、RBAC 权限控制、MFA、审计日志、OAuth/OIDC/SAML/LDAP 协议支持——这些事情单个看都不复杂，但合在一起，就是一个需要专门团队维护数月的基础设施。

Casdoor 把这一切打包成一个开箱即用的平台。你不需要懂 OAuth 的授权码流程，不需要自己实现 SAML 断言解析，不需要手写 RBAC 的权限校验逻辑。你只需要部署一个 Casdoor 实例，在 Web 控制台里点几下，在代码里加几行 SDK 调用——一套完整的身份系统就上线了。

但 Casdoor 真正让人兴奋的地方，是它对 AI Agent 时代的拥抱。在 Logto 用独立的 MCP Server 让 AI"更容易使用身份系统"的同时，Casdoor 选择了一条更激进的路——把**MCP 网关和 A2A 协议直接内嵌到核心架构里**，让身份系统本身成为 AI 生态的一部分。
