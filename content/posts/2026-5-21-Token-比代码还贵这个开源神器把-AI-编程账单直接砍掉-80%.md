**大家好，我是雷达君。**

你经历过这种场景吗：用 Claude Code 重构了一个下午，代码改得漂亮、测试全部跑过、Git 提交一气呵成。心情正美，顺手看了一眼 API 用量——**一个下午烧掉了 6000 多万 token**，账户余额直接报警。

明明只改了几十个文件，怎么耗了这么多？

翻了一遍对话历史你才发现，真正的凶手根本不是写代码的环节，而是那些**命令输出**：

- `npm install`跑一次，依赖树打印几百行
- `cargo test`执行完，99%的通过用例全是绿字刷屏
- `git status`列出一堆 untracked 文件，AI 全盘塞进上下文

这些内容被原封不动地喂给了 LLM。而 AI 真正需要的核心信息，可能只占 5%，剩下 95%全是**日志噪音**。

更扎心的是，这些命令的输出是按 token 计费的。同样多的钱，本可以用来做更有价值的事情，却被浪费在了模型根本不需要的日志上。

今天雷达君安利的这个开源项目，定位就是来解决这个问题的。它是一个高性能的 CLI 代理，在命令输出到达 LLM 之前做一轮智能压缩，把冗余部分过滤掉，只把真正关键的信息递给 AI。

GitHub 上，它的 Star 数已经达到了约**4.2 万**，中文社区普遍评价其为“Token 焦虑终结者”。

它就是——**RTK**（Rust Token Killer）。

## 01 ｜核心逻辑

RTK 的设计思路其实非常朴素，甚至是有点“反直觉”——解决问题的角度，不是你该换什么模型，也不是你该少执行什么命令，而是**既然模型最终只需要看输出的 10%，那为什么要把剩下的 90%塞进账单里？**

RTK 作为一个中间层代理，它的处理流程是这样的：

你照常用 Claude Code、Cursor 或 Codex 这些 AI 编程工具，它们执行命令、读取输出——这是它们惯常的路子。但在 RTK 加入之后，命令的输出不是直接交给模型，而是先经过一条处理链路，由 RTK 做一轮**过滤、聚合、截断、去重**的处理，再把精简过的高密度信息传给模型。

对模型来说，它看到的依然是简洁清晰的执行结果，整个压缩过程完全无感。但 Token 的消耗，已经从原始输出的几千甚至上万，压缩到了几百或几十。

RTK 的官方 README 给出了一组实测数据：在一个 30 分钟的 Claude Code 会话里，原始 Token 用量约为 118,000，经过 RTK 处理后降至约 23,900，整体降幅达到了**约 80%**。

这些节省都来自这里：

| 命令场景                  | 频次  | 原始 Token | RTK 后 | 节省幅度 |
| :------------------------ | :---- | :--------- | :----- | :------- |
| `ls` / `tree`             | 10 次 | 2,000      | 400    | **-80%** |
| `cat` / `read`            | 20 次 | 40,000     | 12,000 | **-70%** |
| `grep` / `rg`             | 8 次  | 16,000     | 3,200  | **-80%** |
| `git status`              | 10 次 | 3,000      | 600    | **-80%** |
| `git diff`                | 5 次  | 10,000     | 2,500  | **-75%** |
| `git add/commit/push`     | 8 次  | 1,600      | 120    | **-92%** |
| `cargo test` / `npm test` | 5 次  | 25,000     | 2,500  | **-90%** |
| `pytest`                  | 4 次  | 8,000      | 800    | **-90%** |

_数据来自 RTK 官方 README，基于中等规模 TypeScript/Rust 项目估算_

从数据不难看出，那些原本最消耗 Token 的命令场景——比如测试、长文件读取和跨文件搜索——恰好是 RTK 压缩收益最高的场景。而这些场景，恰恰是 AI 编程中使用频率最高的命令类别。Token 消耗最大的不是模型推理本身，而是这些命令的“副产品”。

**RTK 不是在省推理，是在帮你省那些模型本来就不应该看的钱。**

## 02 ｜硬核功能盘点

🔗 **GitHub 地址**：https://github.com/rtk-ai/rtk

**1. 四大压缩策略：智能筛子，只留信号**

RTK 的核心压缩能力，可以拆解成四个独立的过滤策略，它们会根据命令类型自动组合：

- **智能过滤：** 去掉 LLM 完全不需要的信息——ANSI 颜色控制码、进度条动画、bootstrap 话术、装饰性分隔线。`git push`原本有十几行输出，RTK 过滤后只保留一行"ok main"核心结果，200token 直接压缩到 10token。
- **分组聚合：** 把同类输出归类汇总，避免逐条罗列。`ls`列出 100 个文件时，原始输出 100 行，RTK 会聚合为"src/目录下 45 个.java 文件，test/目录下 20 个测试文件"这样的紧凑摘要，2000token 压到 400token。
- **截断保留：** 长日志只留关键部分。`cargo test`刷出 25000token 的输出，RTK 仅汇总通过的测试数量，完整保留失败用例的错误堆栈和上下文，压缩到 2500token，节省比例 90%。
- **重复模式合并：** 识别重复的输出结构并折叠。`cargo check`的编译日志中反复出现的"Compiling..."、容器日志的时间戳前缀等重复内容，都会合并为单一示意。

**2. 内置 tee 机制：精简版不够用？完整日志在本地**

RTK 在处理命令输出时，会用`tee`机制做一个双通道处理：压缩后的精简输出传递给模型；同时，完整的原始输出自动保存到本地文件中。当 AI 觉得压缩后的信息不足以做出准确判断时，可以直接去读取本地保存的完整日志文件，无需重新执行命令。

这个设计解决了“压缩会不会丢信息”的核心顾虑。RTK 给模型的是精简版，但给开发者的是一条可回溯的完整数据链。

**3. 100+命令全覆盖，Rust 原生、零依赖**

RTK 对超过 100 种常见开发命令做了针对性的过滤与压缩策略，覆盖日常开发最常用的工具链：文件操作（ls、cat、grep、find）和 Git 全家桶（status、diff、log、add、commit、push）都能被 RTK 覆盖。它用 Rust 写成，单一二进制文件，无任何外部依赖，启动延迟小于 10ms，内存占用不到 5MB。

**4. 原生支持 12+款 AI 编程工具**

RTK 目前支持 Claude Code、Cursor、Codex、Gemini CLI、Windsurf、Cline、Roo Code、OpenCode、OpenClaw、Aider、GitHub Copilot CLI、Augment 等 12 款主流 AI 编程工具。

不同的 AI 工具之间，RTK 的处理机制略有差异，主要体现在命令路由层的实现方案上：

| AI 工具                               | 原生 hook 支持      | 可用模式                 | 使用建议                    |
| :------------------------------------ | :------------------ | :----------------------- | :-------------------------- |
| **Claude Code / GitHub Copilot**      | ✅ 原生支持         | Auto‑rewrite（自动重写） | 体验最佳，无手动干预        |
| **Codex / OpenCode / Cursor / Aider** | ✅ 支持             | Auto‑rewrite（自动重写） | init 命令会自动配置         |
| **OpenClaw / Gemini CLI**             | ✅ 支持             | Auto‑rewrite（自动重写） | -                           |
| **其他 MCP 协议工具**                 | ⚠️ 部分支持         | 模式混合 / 退回到手动    | 参考官方对应章节配置        |
| **原生 Windows（非 WSL）**            | ❌ 不支持 bash hook | CLAUDE.md 注入提示词     | 推荐使用 WSL 以获得最佳体验 |

_支持工具列表及模式来源：RTK 官方 README 与 10+份真实评测中的工具兼容性验证。_

_现实摩擦提醒：hook 机制通常只作用于 Bash 工具调用。如果某些 AI 产品内置了 Read、Grep、Glob 这类不走 bash 的路径，它们不会自动被 RTK 重写。官方建议在这些场景下改用 shell 的 cat/head/tail、rg/grep、find，或者显式调用`rtk ...`来确保压缩生效。_

**5. 跨平台支持：macOS/Linux/Windows 全兼容**

RTK 覆盖 macOS、Linux、Windows 三大主流操作系统。Windows 用户使用 WSL 可获得与 macOS/Linux 一致的完整体验；若使用原生 Windows，RTK 会通过 CLAUDE.md 注入模式退守工作，功能正常但命令不会自动重写。

## 03 ｜快速上手指南

RTK 的安装和初始化非常快，推荐 Homebrew 一键安装：

**方式 ①：Homebrew（macOS/Linux 推荐）**

```bash
brew install rtk
```

**方式 ②：一键安装脚本（Linux/macOS 通用）**

```bash
curl -fsSL https://raw.githubusercontent.com/rtk-ai/rtk/refs/heads/master/install.sh | sh
```

**方式 ③：Cargo 源码安装（Rust 开发者）**

```bash
cargo install --git https://github.com/rtk-ai/rtk
```

> **注意**：crates.io 上已存在另一个同名 Rust 包。如果`rtk gain`命令执行失败，说明装错了版本，请用`cargo install --git`的方式重新安装。

**初始化：**
安装完成后，根据你使用的 AI 工具执行对应的初始化命令（-g 表示全局生效）：

```bash
# Claude Code / GitHub Copilot
rtk init -g

# Codex
rtk init -g --codex

# OpenCode
rtk init -g --opencode

# Cursor
rtk init -g --agent cursor

# Gemini CLI
rtk init -g --gemini
```

重启 AI 编程工具后，hook 即可生效。后续执行 git status、cargo test 等命令会自动被 RTK 重写，无需额外操作。

**验证安装：**

```bash
rtk --version
rtk gain
```

`rtk gain`命令会显示当前会话中已节省的 token 统计信息。

## 04 ｜典型场景和避坑

**适合谁用？**

- **深度依赖 Claude Code、Cursor、Codex 等 AI 编程工具、觉得 API 账单越来越贵的重度用户**：每月省下的 API 费用远超出你装这个工具所花的三分钟。
- **从事大型项目、频繁跑测试/构建/日志搜索的开发者**：每当 cargo test 跑一次，RTK 就帮你省掉 25000token；类似的节省在你每天几十次的测试循环里持续累积。
- **预算敏感的个人开发者或小团队**：省下的 Token 就是实打实的钱。

**不适合/需要谨慎的场景：**

- **极少数场景可能需要完整上下文来做复杂调试**：当 AI 需要阅读那些被 RTK 截断的多余字段来做深度诊断时，RTK 可能会丢掉一点 AI 实际还需要参考的信息。但官方提供了`tee`机制的本地原始日志完整备份，支持随时回查。
- **不走 bash 的 AI 内置工具（Read/Grep/Glob）**：不会被 RTK 自动重写。官方建议在这些场景下改用 shell 的原生命令或显式调用`rtk`。
- **追求零运维、完全托管 AI 开发体验的用户**：RTK 本身几乎没有维护成本，但它毕竟是一个主动安装的工具。

## 05 ｜横向对比

类似 RTK 定位的产品不多，但市场里确实存在几个相近思路的工具。这些工具和 RTK 是**不同层次、可叠加使用**的关系，而不是竞争：

| 维度              | **RTK**                          | **9Router**                           | **Headroom**                   |
| :---------------- | :------------------------------- | :------------------------------------ | :----------------------------- |
| **核心定位**      | 命令输出压缩代理                 | AI 订阅/API 统一路由+多账号轮询       | 全上下文端对端压缩             |
| **解决的问题**    | 命令输出中 95%是噪声，浪费 Token | 多个 AI 提供商/订阅的额度破碎化       | 在更全局的范围压缩上下文尺寸   |
| **降本机制**      | 过滤/聚合/截断/去重 → 压缩输出   | 三层回退+多账号轮询+RTK 整合          | 对整个会话的上下文再做一轮通缩 |
| **与 RTK 的关系** | —                                | RTK 可作为其中一项 token 优化手段引入 | 可与 RTK 串联，实现叠加压缩    |
| **节省幅度**      | 60-90%（命令输出侧）             | 取决于订阅组合                        | 叠加 RTK 时可进一步压缩        |

RTK 和 9Router 可以被理解为**不同层次的成本优化工具**：RTK 让你的每一块钱花得更值——原本要用一整桶水（原始输出）才能浇透模型，现在你一瓢浓缩内容（压缩输出）就能搞定。而 9Router 的解决思路是让你花更少的钱买到同样的水量——通过路由方案选择更便宜的提供商。最理想的情况是两个一起用，把水费和桶一起省。

## 06 ｜总结

RTK 解决的是一个很朴素但很致命的问题：**在 AI 编程场景里，Token 的消耗大头不是核心推理，而是命令输出的副产品。**

据 GitHub 项目页的官方数据，一场 30 分钟的 Claude Code 会话，原始 token 用量约 11.8 万，其中至少有 80%是命令输出中的噪声。通过 RTK 的智能压缩，这些 token 在模型读取之前就已完成过滤，而且整个过程对模型和你的使用习惯完全透明。

这个结论其实挺冲击的：你每个月给 AI 编程序掏的钱里，很大比例只是花在了让模型看一遍“npm 安装了几百行依赖树”这类机器信息上——而 RTK 做的事，就是在你看不见的地方把这些噪音挡在门外，把节省的每一分钱还给真正有价值的对话和推理上。

Token 比代码还贵的时代，用好 RTK，就是最省心的一种回报。
